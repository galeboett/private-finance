# Private-Finance: Architectural Plan for Problems B–E

**Scope:** Mistake correction (B), drill-down navigation (C), interactive dashboards (D), and low-friction data import (E).
**Constraints honored:** Local-first, self-hosted, FastAPI + SQLite backend, React/Vite/TypeScript frontend, two-person collaboration model.
**Relationship to existing plan:** These designs slot into Phases 3–6 of the remediation roadmap. Problem C and D depend on the Phase 3 frontend re-architecture; Problem B introduces schema changes that should land *before* the frontend work; Problem E extends Phase 6 (import robustness).

## 0. Cross-Cutting Architectural Decisions

Before the per-problem designs, three decisions that all four problems share:

### 0.1 Frontend state & routing foundation (prerequisite from Phase 3)

The 3,568-line `App.tsx` monolith cannot support drill-down navigation, shareable filter state, or undo toasts cleanly. The plan below assumes the Phase 3 decomposition adopts:

- **React Router** (or TanStack Router) with URL-driven state. Every filter, date range, and drill-down context is encoded in the URL. This is the single most important decision for Problem C: it makes every investigative view bookmarkable, back-button friendly, and shareable between the two collaborators.
- **TanStack Query** for server state (caching, invalidation after mutations — critical for undo).
- **A thin global UI store** (Zustand) only for ephemeral UI state: selection sets, undo toast queue, modal state.

## Problem B — Fixing Mistakes: Bulk Edits, Deletions, and Undo

### B.1 Approach: Operation Journal + Inverse Operations (not event sourcing)

Full event sourcing is overkill for a two-user SQLite app. Instead, use a **mutation journal**: every write (single or bulk) is recorded as an *operation* containing the before-image of each affected row. Undo replays the before-images. This gives:

- Bulk edit / bulk delete as first-class operations
- Undo (single-level and history-based) with no architectural ceremony
- A human-readable activity log ("who changed what, when") — valuable for the two-person collaboration model
- Soft deletes for free

### B.2 Data schema

```sql
-- Journal of logical operations (one row per user action, even if it touched 500 txns)
CREATE TABLE operations (
    id            TEXT PRIMARY KEY,          -- uuid4
    kind          TEXT NOT NULL,             -- 'bulk_update' | 'bulk_delete' | 'update'
                                             -- | 'delete' | 'create' | 'import' | 'undo'
    entity_type   TEXT NOT NULL,             -- 'transaction' | 'account' | 'rule' …
    actor         TEXT NOT NULL,             -- collaborator identity (no hardcoded names —
                                             -- reuse the fix from the earlier hotfix)
    description   TEXT NOT NULL,             -- "Recategorized 43 transactions → Groceries"
    created_at    TEXT NOT NULL,
    undone_by     TEXT REFERENCES operations(id),  -- set when this op is reverted
    undo_of       TEXT REFERENCES operations(id)   -- set when this op IS a revert
);

-- Row-level before/after images belonging to an operation
CREATE TABLE operation_changes (
    id            INTEGER PRIMARY KEY,
    operation_id  TEXT NOT NULL REFERENCES operations(id),
    entity_id     TEXT NOT NULL,
    before_json   TEXT,          -- NULL for creates
    after_json    TEXT           -- NULL for deletes
);

-- Soft delete on transactions
ALTER TABLE transactions ADD COLUMN deleted_at TEXT;  -- NULL = live
```

Rules:
- All read queries filter `deleted_at IS NULL` (enforce in one shared query helper, not ad hoc).
- A nightly/opt-in **purge** hard-deletes rows soft-deleted > 90 days ago (configurable), keeping the DB lean.
- `before_json`/`after_json` store only the changed columns plus the primary key — keeps journal size small on wide bulk edits.
- All journal writes happen inside the same SQLite transaction as the mutation (`contextlib.closing` + explicit `BEGIN`), so the journal can never disagree with the data.

### B.3 API surface

```
POST   /operations/bulk-update      { entity_type, ids[], patch:{category_id,…} }
POST   /operations/bulk-delete      { entity_type, ids[] }
POST   /operations/{id}/undo
GET    /operations?limit=50&entity_type=&actor=       # activity log
GET    /operations/{id}                               # detail w/ per-row diffs
```

Undo semantics:
- Undo of an update/bulk-update ⇒ write `before_json` back to each row.
- Undo of a delete ⇒ clear `deleted_at`.
- Undo of a create/import ⇒ soft-delete the created rows.
- **Conflict guard:** if any affected row was modified by a *later* operation, return `409` with the list of conflicted rows; UI offers "undo only unconflicted rows."
- Undo is itself journaled (`undo_of` set), so it appears in the activity log and is itself undoable (redo).

### B.4 UX/UI

**Selection + bulk action bar (transactions table):**
1. Checkbox column; shift-click for range select; header checkbox = "select all on page," with a banner offering "Select all 1,238 matching current filter" (server-side selection by filter, not by ID list, for large sets).
2. When ≥1 row selected, a **sticky bottom action bar** slides up: `43 selected — [Change category] [Change account] [Add tag] [Edit date] [Delete] [✕]`.
3. Each bulk action opens a small popover, not a modal — keeps the table in view.
4. On completion: **undo toast** bottom-left, persisting 10 seconds: *"Recategorized 43 transactions → Groceries — Undo"*. Clicking Undo calls `POST /operations/{id}/undo` and invalidates queries.

**Activity page (`/history`):**
- Reverse-chronological list of operations: description, actor, timestamp, affected-row count, and an **Undo/Restore** button on each entry (disabled with a tooltip if conflicted).
- Expanding an entry shows per-row diffs (before → after) for auditability between the two collaborators.

**Trash view:**
- `/transactions?view=trash` lists soft-deleted rows with Restore and Delete-forever actions. Delete-forever requires a confirm dialog naming the count.

### B.5 Implementation steps

1. Migration: add `operations`, `operation_changes`, `deleted_at` column; backfill nothing.
2. Build `mutation_log.py` service: `journal(op_kind, changes) → operation_id` used by *every* existing write path (single edits included — retrofit).
3. Add bulk endpoints + undo endpoint with conflict detection; pytest coverage for undo-after-conflict, undo-of-undo, import-undo.
4. Frontend: `useSelection` hook, `BulkActionBar`, `UndoToast`, `ActivityLog` page.
5. Wire the existing import pipeline to journal each import as one operation → **imports become undoable**, which pairs perfectly with the BUG-02 idempotency work.

**Completion note (2026-07-12):** All five steps are now implemented for ordinary finance-domain changes. The journal supports mixed entity types within one user action, so category merges, account cleanup/deletion, transfers, rules, splits, allocations, holdings, app-data imports, and both transaction import paths can be reviewed and undone as coherent operations. Shared frontend selection, bulk-action, and undo-toast components now provide the same recovery behavior across related editing experiences. Intentional safety boundaries remain outside ordinary undo: authentication changes, read-only previews, delete-forever actions, and database backup/restore. Those flows use confirmation, session revocation, or pre-restore safety copies instead of the mutation journal.

## Problem C — Drill-Down: From Totals to Transactions

### C.1 Approach: One canonical filter model, everywhere

The core architectural idea: **every number on every dashboard is a query with a filter, and clicking that number navigates to the transactions view with that exact filter applied.** No bespoke drill-down code per chart — one shared mechanism.

### C.2 The filter model

```ts
// lib/filters.ts
interface TxnFilter {
  accounts?: string[];
  categories?: string[];       // supports category tree: parent implies children
  tags?: string[];
  dateFrom?: string;           // ISO date
  dateTo?: string;
  amountMin?: number;
  amountMax?: number;
  direction?: 'inflow' | 'outflow';
  search?: string;
  view?: 'live' | 'trash';
}
```

- Serialized to/from URL query params (`/transactions?categories=groceries,dining&dateFrom=2026-06-01…`).
- The backend accepts the identical shape on `GET /transactions` and on every aggregation endpoint, so **the sum shown in a chart and the rows shown after drill-down are guaranteed to be computed from the same predicate** — the classic "chart says $4,102 but the table sums to $4,088" bug becomes structurally impossible.

### C.3 Interaction patterns (uniform across all pages)

| User action | Result |
|---|---|
| Click a stat tile (e.g., "June spending: $4,102") | Navigate to `/transactions` with that tile's filter |
| Click a bar/segment/pie slice | Same, filter = that dimension value + current date range |
| Click a point/region on a time-series | Filter = that date bucket |
| Click a row in a category-breakdown table | Expand in place to sub-categories (level 1), second click navigates to transactions (level 2) |
| Hover any clickable element | Cursor pointer + tooltip footer: "Click to view 87 transactions" |

**Breadcrumb / filter-chip bar** at the top of the transactions view shows the active filter as removable chips: `Groceries ✕ · Jun 2026 ✕ · Chase Checking ✕`. Removing a chip broadens the query. A "Save this view" affordance can come later.

**Peek drawer (fast path):** For light investigation, clicking a chart element opens a right-side **drawer** with the top ~20 matching transactions and a "Open full view →" link. This keeps the user's dashboard context intact for quick "what was that spike?" questions — the most common drill-down motive — while the full navigation remains one click away.

### C.4 Backend aggregation endpoints

```
GET /aggregate/by-category?{TxnFilter}   → [{category_id, total, count}]
GET /aggregate/by-account?{TxnFilter}
GET /aggregate/timeseries?bucket=day|week|month&{TxnFilter}
```

All implemented in `services/aggregation.py` against one shared `WHERE`-clause builder that also powers `GET /transactions` (see C.2 guarantee). Add pytest property-style tests asserting `sum(aggregate) == sum(list)` for random filters.

### C.5 Implementation steps

1. Build `filters.ts` (model + URL codec) and the backend `WHERE`-builder; unit-test round-tripping.
2. Refactor transactions endpoint + table to consume `TxnFilter`; add chip bar.
3. Add aggregation endpoints; migrate dashboard numbers to use them.
4. Add click-through wiring: a single `useDrillDown(filterFragment)` hook returning `{onClick, href}` used by every tile/chart element.
5. Add the peek drawer.

## Problem D — Interactive Dashboards (Net Worth Example)

### D.1 Data model: daily snapshots

Charts like the reference image need a stable time series, independent of transaction edits:

```sql
CREATE TABLE net_worth_snapshots (
    snapshot_date  TEXT NOT NULL,          -- ISO date
    account_id     TEXT NOT NULL,
    balance        REAL NOT NULL,
    source         TEXT NOT NULL,          -- 'import' | 'computed' | 'manual'
    PRIMARY KEY (snapshot_date, account_id)
);
```

- One row per account per day (per-account granularity enables the account-level drill-down from Problem C: click the chart → see which account moved).
- Populated by: (a) imports that carry statement balances (relies on the BUG-03 snapshot-date fix — extraction now returns real dates), (b) a computed fallback that derives balances by rolling transactions forward from the last known balance, (c) manual entry for accounts with no feed (house value, etc.).
- Gaps are **forward-filled at query time**, never stored, so late-arriving imports self-heal the series.

### D.2 API

```
GET /snapshots/networth?from=&to=&bucket=day|week|month
     → { series: [{date, total, byAccount:{…}}], … }

GET /snapshots/networth/stats?from=&to=
     → { start, end, change, changePct, min, max, minDate, maxDate,
         bestDay:{date,delta}, worstDay:{date,delta} }
```

The stats endpoint powers the drag-selection panel — the frontend never computes financial stats itself, keeping one source of truth.

### D.3 Chart UX (matches and extends the reference image)

**Layout (top → bottom):**
1. **Header stats row** — Account value, total gain/loss, period change (these become drill-down tiles per Problem C).
2. **Main area chart** — gradient-filled line, right-side y-axis labels, hover tooltip pinned above the point (`July 10 · $333,240.68` as in the image).
3. **Period selector** — segmented control: `1M · 6M · 1Y · Max` (persist choice in URL: `/net-worth?period=6M`).
4. **Brush selection**: click-and-drag on the chart shades the selected region and opens a **stats strip directly under the chart**:

```
Jun 18 – Jul 2  ·  +$6,214 (+1.9%)  ·  High $331,970 (Jul 1)  ·  Low $324,880 (Jun 19)  ·  [View transactions →] [✕]
```

   - Drag handles on both edges of the shaded region allow refining the range.
   - `Esc` or ✕ clears the selection.
   - "View transactions →" applies `dateFrom/dateTo` via the Problem C filter model — the two features compose.
5. **Bucketing rule:** 1M → daily points; 6M → daily; 1Y → weekly; Max → monthly. Keeps Max renders fast and readable.

**Implementation choice:** Recharts `AreaChart` + `ReferenceArea` for the drag-shade (mouse-down/move/up handlers set the area bounds), with the existing `Brush` component as a fallback if custom drag proves fiddly. Recharts is already in the artifact-supported stack and is sufficient; avoid adding D3 unless a later phase demands it.

**Same pattern generalizes:**
- **Spending page:** stacked bar by month with category legend toggles; drag-select months → stats strip + drill-down.
- **Accounts page:** small-multiple sparklines per account, each clickable into the full net-worth chart filtered to that account.

### D.4 Implementation steps

1. Migration for `net_worth_snapshots`; backfill from historical imports where balances exist, computed-fallback elsewhere.
2. `snapshots.py` router with series + stats endpoints; pytest for forward-fill and bucket edges (month boundaries, single-point ranges).
3. `NetWorthChart` with hover tooltip and period selector.
4. Add drag-selection + `BrushStatsPanel`; wire "View transactions" through the shared filter model.
5. Port the pattern to spending/accounts pages.

## Problem E — Convenient, Private Data Import (10–20 Accounts)

### E.1 The landscape, honestly assessed

| Option | Convenience | Privacy | Effort | Notes |
|---|---|---|---|---|
| Manual CSV/OFX download (current) | ✦ | ✦✦✦✦✦ | Low | 10–20 logins monthly is the pain point |
| **SimpleFIN Bridge** | ✦✦✦✦ | ✦✦✦✦ | Medium | Read-only aggregation designed for exactly this use case; user grants access once, app pulls a token-scoped, read-only feed. Credentials never touch your app |
| Plaid / MX / Finicity direct | ✦✦✦✦✦ | ✦✦ | High | Excellent coverage, but a third party sees all data and dev-account terms are awkward for personal use |
| Self-hosted scraping (e.g., headless browser per bank) | ✦✦✦ | ✦✦✦✦✦ | Very high | Fragile, breaks with every bank UI change, MFA is painful; not recommended as the core strategy |
| Email statement parsing | ✦✦✦ | ✦✦✦✦ | Medium | Good *supplement* for banks that email statements/alerts |
| Watch-folder auto-ingest | ✦✦✦ | ✦✦✦✦✦ | Low | Doesn't remove logins, but removes every step *after* download |

### E.2 Recommended architecture: three tiers, user chooses per account

**Tier 1 — Watch-folder "Import Inbox" (build first, lowest risk):**
The app watches a configured directory (e.g., `~/Downloads/finance-inbox/`). Any file dropped there is fingerprinted (SHA-256, journaled to prevent re-ingest), auto-matched to an institution by an **importer registry** (filename pattern + header sniffing), parsed, deduplicated via the existing idempotency keys (BUG-02 fix), and staged. The user's monthly ritual collapses to: log in → click download → done. No copying, renaming, uploading, or picking an account in the app.

```
POST /imports/inbox/scan          # manual trigger; also a background watcher task
GET  /imports/pending             # staged batches awaiting review
POST /imports/{batch_id}/confirm
POST /imports/{batch_id}/discard
```

**Import Review screen:** each pending batch shows institution, detected account, date range, row count, `N new / M duplicates skipped`, and a preview table with anomaly flags (unparseable dates, unknown categories). Confirm journals the whole batch as one operation → **fully undoable via Problem B.**

**Tier 2 — SimpleFIN Bridge connector (the automation tier):**
For accounts at supported institutions, offer a per-account "Connect via SimpleFIN" flow. Architecture safeguards for a privacy-first posture:
- Access tokens stored **encrypted at rest** in SQLite (key from an env/OS-keychain secret, never in the repo — consistent with the machine-specific-paths fix).
- Pull-only, on-demand or scheduled fetch from the FastAPI backend; no inbound connections, no webhooks.
- Fetched data flows into the *same* staging/review pipeline as Tier 1 — one dedup and confirmation path regardless of source.
- Clearly label in the UI which accounts flow through a third party, so the privacy trade-off is explicit and per-account. Institutions the user considers sensitive can stay on Tier 1 forever.

**Tier 3 — Email parsing (optional, later):**
A dedicated mailbox (or IMAP filter) receives bank statement/alert emails; a poller extracts attachments/amounts and drops them into the same inbox pipeline. Good for the long tail of small accounts.

### E.3 Importer registry design (Phase 6 synergy)

```
backend/app/services/importers/
├── base.py            # Importer protocol: matches(file) -> bool, parse(file) -> [TxnDraft]
├── registry.py        # ordered matcher; falls back to "generic CSV w/ column-mapping UI"
├── chase_csv.py
├── fidelity_csv.py
├── venmo_csv.py       # existing (with the monkeypatch-tested config)
└── generic_csv.py     # user maps columns once; mapping saved per fingerprint pattern
```

The generic importer's saved column mappings mean even unrecognized banks require setup exactly once.

### E.4 Implementation steps

1. Build the staging tables (`import_batches`, `import_rows`) + fingerprint journal; route the *existing* manual upload through staging/review (unifies everything early).
2. Importer registry + 3–4 institution modules covering the highest-volume accounts; generic CSV mapper for the rest.
3. Watch-folder background task (FastAPI lifespan task + `watchfiles`); Import Inbox UI.
4. SimpleFIN connector: token setup UI, encrypted storage, scheduled fetch, per-account source labeling.
5. (Optional) Email poller.

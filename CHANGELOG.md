# Changelog

## Phased game-plan implementation — July 13, 2026

- Phase 0 gates and hygiene: the Import Inbox now defaults outside the repository at `~/PrivateFinance/import-inbox`; both `PF_IMPORT_INBOX` and the older `PF_IMPORT_INBOX_DIR` override are supported; package stores, databases, logs, pytest temp data, and `backend/data/` are ignored; Git history was audited without finding committed statement/data files; and the pre-merge pytest plus production-build gate is documented.
- Phase 0 frontend shell: API access and route exports were extracted from `App.tsx`, feature/shared-component directories were established, and Vitest coverage now protects transaction-filter URL round-tripping.
- Phase 1 sign architecture: per-account/preset sign profiles are persisted, journaled, undoable, and preserved across account merges. Credit-card and payroll plausibility checks prompt on contradictory files without silently changing signs, saved choices apply automatically, and later anomalies remain visible in Import Review.
- Import Review and sign prompting now live under `features/imports/`, reducing `App.tsx` while establishing the strangler pattern for later phases. The canonical sign contract and cleanup guidance are documented in `docs/amount-signs.md`.
- Phase 2 duplicate review: Review now shows candidate/original transactions side by side with differing account, reference, date, amount, description, category, notes, labels, and import-source fields highlighted. Remove-new, keep-both, replace-old, and bulk exact-match resolution are journaled as one undoable operation; replace-old preserves user categories, notes, labels, and splits.
- Duplicate and transfer review moved into `features/review/`; `App.tsx` is 5,509 lines, down from 5,528 at the end of Phase 1 and 5,638 at the start of the phased plan.
- Phase 3 transfer hardening: confirmed transactions can now be matched after both sides arrive, brokerage and retirement ACH transfers use a seven-day window, and confirmed credit-card payment links power per-card payment verification with warnings for stale unmatched payments.
- Phase 3 account balancing: imported running balances create statement checkpoints automatically, existing running balances are backfilled at startup, and manual statement balances are journaled and undoable. Account pages show reconciled/off-by status, an exact investigation date range, and card-payment verification. Checkpoints survive account merges and are covered by account deletion/undo. The account page, reconciliation badge, and payment verification panel live under `features/`; `App.tsx` is 5,506 lines.
- Import compatibility: compact brokerage position exports with report metadata above a `Symbol / Qty / Price / Market Value` header are now detected automatically. Position and cash rows import as holdings, summary totals are excluded, and the snapshot date is read from the export filename.
- Review rules now support categoryless Card payment and Transfer classifications. Single and bulk confirmation no longer require a category for those types; saved rules can classify and confirm them without creating a spending category; and changing or confirming a transfer/payment clears any stale category to prevent payment double-counting. Existing rule tables are upgraded in place without losing saved rules.
- Phase 5 net-worth data management: dated acquisition, quantity, and cost-basis details now live in journaled `holding_lots` records. Fidelity-compatible exports populate lots when acquisition/basis columns are present, refreshed imports replace only imported lots, and manual lots remain intact. Holdings show aggregate basis, unrealized gain/loss, and oldest-lot age. A shared canonical-sign manual transaction form is available from account pages and Net Worth for asset accounts; both lot and transaction creation support Activity undo. `HoldingsPanel`, `ManualTransactionForm`, and delete confirmation are extracted components, reducing `App.tsx` to 5,390 lines.
- Position exports that provide aggregate cost basis without acquisition dates now retain that basis on the holding snapshot. Scan Inbox restages an already-committed positions file only when it can fill missing basis, then returns to normal duplicate skipping after the refresh is confirmed.
- Fidelity `Portfolio_Positions` imports now tolerate Fidelity's lowercase header variants, route Individual/HSA/Amazon 401(k)/BrokerageLink rows into three logical accounts, ignore the duplicate Amazon BrokerageLink summary, and preserve FDRXX/FCASH money-market holdings. A one-time journaled migration repairs the existing Fidelity snapshot, its net-worth totals, the HSA last four, the 401K account type, and the duplicate BrokerageLink account.
- Phase 4 refund matching: positive money-in rows can now be suggested or manually linked to an original expense, including multiple partial refunds. Matching favors the same account, merchant/description overlap (including Venmo notes), compatible amounts, and a 90-day window; confirmed transfers and refunds cannot claim the same rows. Confirm/dismiss/link/unlink actions are journaled and undoable, over-refunds require explicit confirmation, and confirmed refunds inherit the expense category without changing signed reporting totals. Review comparisons, transaction-detail linking, `↩ refunded $X` ledger badges, and a bookmarkable **Has refund** filter live in extracted refund components. Shared controls were also extracted, reducing `App.tsx` from 5,390 to 5,371 lines.
- Refund matching performance and relevance were tightened after large ledgers produced thousands of weak matches. The finder now uses indexed merchant/date/amount candidates, replaces stale open suggestions, excludes payment/transfer/payroll-like rows, and persists at most the 25 highest-confidence matches. The manual picker applies the same relevance rules for its selected expense, returns at most 50 candidates, and debounces search requests.
- Iteration 3 Phase 6 verification and classification correctness: card-payment warnings now require explicit payment context and reject fee, interest, return, benefit, reward, and protection phrases; confirmed types override heuristics; and **Not a payment** dismissals persist, appear in Activity, and support Undo. Reclassifying a payment-like row creates the dismissal inside the same operation. Saved rules can be applied and confirmed on one matching row without a batch preview, the reusable save-rule control is available in both Review and the account ledger, transaction mutations refresh cross-view data through one shared version signal, and left-nav account clicks clear investigation filters. Rule panels were extracted under `features/rules/`; `App.tsx` remains 5,378 lines, unchanged from the phase start.
- Iteration 3 Phase 7 ledger-wide duplicate scan: Review can now scan existing ledger history for exact, cross-source history/CSV, probable-description, and mirrored-sign duplicate pairs. Confirmed transfer/refund rows and previously reviewed **Keep both** pairs are excluded; old keep-both audit decisions migrate into durable pair memory. Exact results can be removed in one undoable bulk action, mirrored sign artifacts have a tailored removal action, and card-account resolutions offer to re-run transfer matching. `LedgerDuplicateScan` is extracted under `features/review/`.
- Duplicate Review now fetches and renders only 25 pairs at a time, with tier filters, queue totals, and Previous/Next controls, preventing large scans from freezing the app. Bulk removal is restricted to the strongest safe-reimport cases: matching source references or an exact categorized-history/bank overlap. Opposite-sign wording now warns that a pair may instead be a legitimate refund or reversal, and the action explicitly says it removes the positive copy.
- Safe duplicate reimports now support both queue-wide directions: keep established ledger data or apply the newest imported bank facts while preserving the established record's annotations and links. Both actions require a stale-safe confirmation preview that discloses pair/account/source scope and the signed ledger adjustment before one journaled, undoable operation runs.
- Categorized-history replacement now has a constrained exact-filename lineage purge with a stale-safe preview token and typed confirmation. It releases old source fingerprints for a clean reimport, removes dependent staging/refund/duplicate records, preserves unrelated financial data, and is covered by integrity and exact-scope regression tests.
- Activity undo now handles imports and bulk operations with thousands of rows without exceeding SQLite's expression-depth limit. Conflict checks run in bounded chunks, and transaction imports only recalculate net-worth snapshots when the imported rows actually supplied running balances.
- Duplicate scans now treat connected **Keep both** decisions as one reviewed group. This prevents the same cluster of identical same-day transactions from resurfacing as every remaining pair combination, and a new scan clears already-queued pairs covered by those prior decisions.
- Intentional historical full-refund pairs now have a strict, preview-first bulk linker: same account/date/description, equal opposite amounts, expense plus refund, matching category, and the same categorized-history batch. It creates explicit refund provenance without deleting rows or changing signed totals. Exact and probable pairs can also be selected page-by-page for one undoable bulk **Keep both** action; selected exact pairs additionally support previewed removal, while probable bulk deletion remains prohibited. Both confirmation surfaces live in extracted `features/review/` components; `App.tsx` remains 5,368 lines.
- Selected exact and probable Manual-entry/history duplicate pairs can now prefer `transaction history for private finance 7.14.26v2.csv` as the authoritative source. The preview discloses category/type changes, affected accounts, the signed ledger adjustment, rows moving to Trash, and preserved annotation counts. Applying it keeps the established transaction identity and its notes, labels, splits, allocations, and links; copies the authoritative bank facts, source lineage, category, and type; retires the redundant imported row; and remains one Activity-undoable operation. Mixed or differently sourced selections are rejected.
- Authoritative-history eligibility now uses the same resolved-source rule as Duplicate Review: an established row whose obsolete import-batch ID points to a batch that no longer exists is displayed and accepted as **Manual entry**, instead of being incorrectly rejected because the stale numeric ID was non-empty.
- Iteration 3 Phase 8 balance anchoring and external accounts: accounts in automatic mode contribute to net worth only after a statement checkpoint, imported/manual balance, or holdings snapshot supplies a real anchor. Net-worth series, statistics, contributors, sidebar balances, startup disclosure, and the account-level inclusion override now distinguish anchored, explicitly included, and excluded accounts. Untracked accounts are excluded from net worth, imports, spending, and cash flow; unmatched card payments can be linked to a journaled synthetic mirror in an untracked account and undone as one operation. `UnanchoredBanner` and `ExternalPaymentAction` are extracted components. The prior 5,144-line count excluded blank lines; the consistent physical count at that checkpoint was 5,368.
- Duplicate Review now supports the same inclusive Shift+click range selection used by the account ledger, backed by an extracted shared `useSelection` hook with unit coverage. Its bulk-action bar remains sticky beneath the application header while scrolling. `App.tsx` is now 5,339 physical lines (5,119 nonblank), down 39 from the iteration baseline.

## Remediation pass — July 2026

IDs reference `private-finance-evaluation-and-plan.md`.

### Fixed

- Raw CSV transaction dedupe no longer depends on a mutable internal account ID. New normalized fingerprints are account-independent; reliable Bank of America and Venmo references take precedence; matching reference/date/amount rows are skipped even when descriptions change; conflicting reference details are linked and sent to review; and ordinary CSV fingerprints are reconciled during account merges. Duplicate warnings now count only active rows. Categorized-history importing is intentionally unchanged.
- Categorized-history credit-card and Venmo amounts now normalize from the legacy Excel convention (positive charges, negative refunds) into one ledger convention (negative charges, positive refunds). A preview-first, journaled maintenance action repairs legacy imports, including soft-deleted rows, dependent splits/allocations, and Venmo's account type with one Undo operation. Its preview shows each historical cutoff and later direct-CSV range, leaves direct imports unchanged, and warns about date overlap, repeated direct-import references, or likely duplicate account aliases. Future history imports conservatively match existing accounts by institution, last four, and safe label aliases to avoid creating those duplicate account records.
- Deleted-transaction Activity details now show a readable Active → Moved to Trash timestamp instead of blank `deleted_at` values; new deletions also journal the exact deletion timestamp.
- **BUG-01** — Re-importing a CSV that overlaps an earlier import crashed with a 500 (`NameError` in `commit_import`) and rolled the whole import back. Duplicates are now skipped and counted correctly. Regression test: `tests/test_import_commit.py::test_commit_import_skips_duplicates_on_reupload`.
- **BUG-02** — Re-importing a brokerage positions file double-counted net worth. Importing positions for an account/date now replaces that snapshot instead of appending to it.
- **BUG-03** — Brokerage snapshot dates were always recorded as "today". Dates are now parsed from the filename (Fidelity `Jul-04-2026` style, ISO, `MM-DD-YYYY`, `YYYYMMDD`); an explicit `snapshot_date` query parameter on `/api/imports/commit` overrides the filename; when neither is available the import warns and uses today.
- **BUG-04** — Categorized-history imports typed positive deposits into checking/savings as expenses, inflating spending totals. Deposits now default to `income` (with `refund` labels respected).
- **BUG-05** — `/api/imports/commit` did not enforce the file-size limit and non-UTF-8 uploads produced a 500. Size is now checked and undecodable files return a clear 400.
- **BUG-06** — Overview metric tiles silently switched data sources when a period's cash-flow total was exactly $0 (`||` fallback). The fallback now triggers only when cash-flow data is absent entirely.
- **BUG-07** — The Venmo importer hardcoded a personal name; it is now the `PF_VENMO_SELF_NAME` setting, with a neutral From→To fallback when unset. `scripts/restart.ps1` no longer hardcodes machine-specific tool paths; tools resolve from `PATH` with `PF_PYTHON` / `PF_PNPM` / `PF_NODE_DIR` overrides and an npm fallback.
- **BUG-09 / SEC-01** — Backups now use SQLite's online backup API (consistent while the app runs), are constrained to `data/backups/`, and restores validate the file, write an automatic `pre-restore-<timestamp>.sqlite3` safety copy, and dispose pooled connections before swapping the database.
- **SEC-05** — `verify_password` no longer swallows unexpected errors, and hashes are transparently upgraded on login when Argon2 parameters change.

### Changed (requirements update)

- Import Inbox discovery is user-initiated. The app no longer scans the folder on startup or every 30 seconds; files are searched and staged only after the user selects **Scan inbox**.
- Applying a saved rule to existing transactions confirms them by design (requirement changed; `docs/workflow.md` updated to match). Import-time rule matches now also carry the rule's suggested transaction type and remain in review as `suggested`.

The analysis toolbar stays visible while scrolling, and Overview cards are limited to high-level account, cash-flow, and spending context instead of repeating review/import work or duplicating cards inside dedicated tabs.

### Review pass (code review of this remediation)

- Backup and pre-restore safety-copy SQLite connections are now explicitly closed (`contextlib.closing`); `with sqlite3.connect(...)` only scopes transactions, so the previous code leaked file handles that could hold locks on Windows.
- `test_preview_venmo_ignores_crypto_summary_rows` now sets `venmo_self_name` like its sibling tests; it previously asserted self-name-aware phrasing without configuring the name and would have failed.
- Finding IDs above were updated to match the renumbered evaluation plan (the rule auto-confirm item was withdrawn per the requirements change, and the metric-tile fallback is now BUG-06).
